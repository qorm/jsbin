# 属性访问 / 内联缓存性能专项 — 设计评审(初稿)

> 日期:2026-07-19 · 状态:待评审 · 关联:plan.md S5「编译速度专项」、docs/PERF_PLAN.md P3 线
> 铁律:fixtures 不降、macos-arm64 全链 gen1==gen2==gen3、增量参数化(禁大段复制)、
> 内存布局变更原子提交(铁律 4,本专项最大风险点)。

---

## 1. 问题与测量

出厂编译器(asm.js 二进制)自编译 cli.js **25.5s**,node 驱动(gen0)**1.55s**——**~16×**。
bench 三件套(vs Node 24,墙钟):数值 1.6×、**属性访问 ~13×**、字符串 611×。
node 侧编译 profile:fixupAll 9.5%、闭包分析 11%、macho 写出 6%、GC 9.7%——时间**均匀散布**
在编译器 JS 逻辑的属性访问/调用上,被 13× 的属性差距整体放大。已排除:GC(关阈值更慢)、
字符串拼接热循环(不存在)、调用次数(emit32/macho 批量 push,实测收益 ~0)。

**结论:唯一大体量杠杆 = 属性访问路径本身。** 编译速度是表象,所有产物的运行速度同病。

## 2. 现状架构(精确到行)

### 2.1 对象布局(48B 头,`runtime/types/object/index.js:7-31`)

```
+0 type | +8 count | +16 __proto__ | +24 capacity | +32 props_ptr | +40 flags_ptr
props 数组:(key, value) 16B/对,capacity 翻倍增长(P0-3);key 为装箱驻留字符串(P0 驻留)。
```

### 2.2 查找路径 `_object_get`(:299-460)

类型/防御层 ~15-20 指令(tag/null/ptrFloor/Map/Set/AB/Symbol/Proxy 分支)→
阶段A 纯指针扫(每键 load+cmp+add+jmp ≈4-5 指令)→ 阶段B 首字节预滤 + `_object_key_eq`
(每次候选一次 call)→ 命中取值;miss 转原型链。
小对象(5 属性,命中 index 2-3)≈ **30-40 指令/读**。

### 2.3 现有 IC `_object_get_ic`(:1056-1175)

站点缓存**上次命中的下标**(数据段 8B 槽),零 prologue 快路:
tag 检查 → 脱壳+floor → type==OBJECT → 读缓存下标 → 读 count 判越界 → 读 props_ptr 判空 →
按下标取 key 与查询 key 单 cmp 自验证 → 取值 → getter 判定。
**命中 ≈22 指令 + 一次 call 开销**;键指针相等(驻留)保证永不需失效;miss 落 framed 慢路回填。

### 2.4 差距分解(IC 命中 ~22 指令 vs V8 ~3-5)

| 构成 | 指令数 | 性质 |
|---|---|---|
| 防御层(tag/floor/type/count 越界/props_ptr 判空) | ~12 | 每次命中重复支付的**语义冗余** |
| 键自验证 cmp + getter 判定 | ~4 | 必要 |
| 实际取值(下标换算 + load) | ~3 | 本体 |
| call 开销(bl + LR) | ~2-3 | 未内联 |
| 多态站点下标颠簸 | 不定 | 形状不同时缓存互踢 |

**洞察:12/22 的防御指令本质都是「形状证明」——一次形状比较可全部折叠掉。**

## 3. 方案

### 方案 A — 形状(隐藏类)IC(结构性终局)

给对象引入 shape 标识;站点缓存 (shape → 下标);命中 = 读 obj.shape、与缓存 cmp、
按缓存下标直接取值(~6-8 指令,call 保留或消除)。

- shape 可免新头字段起步:**用 props 数组地址等价物或独立 shape 对象指针**;成熟形态是
  属性追加时的 shape 转移表(addA: shape0→shape1...)。
- 威力:防御层全折叠;多态站点稳定(每形状一缓存,可加单槽双路)。
- 风险:**对象头/形状存储的内存布局变更触发铁律 4**(原子改全部遍历站点);
  shape 生命周期与 GC(侧表根区);flags/attrs(flags_ptr@40)与 shape 的交互;
  编译期 key 驻留前提下收益上限 ~3-4×(属性访问),自编译 25.5s → 估 8-12s。
- 工作量:中-大(布局变更 + shape 表 + IC 改造 + 门禁每步回放)。

### 方案 B — 站点直存 (props_ptr, index) 自验证 IC(推荐的第一步)

不动对象布局。站点从「缓存下标」扩为「缓存 props_ptr + 下标」:
命中路径 = 读 obj.props_ptr、与缓存 cmp(对象被 grow 重分配即 miss)→ 读缓存下标处 key、
与查询 key cmp(自验证)→ 取值。**~8-10 指令**,无布局变更、无 shape 概念、自验证天然安全。

- 适用前提:单体/稳定对象(预热后不再 grow)——编译器自身(AST 节点、context、标签表)
  与绝大多数用户对象正是此形态;grow 发生时自然 miss 回填。
- 风险:小(纯 IC 站点槽格式变更 + 快路指令重排;属编译器/asm 增量,不动运行时布局)。
- 收益:属性访问 ~13× → 估 5-7×;自编译 25.5s → 估 15-18s。
- 工作量:小(1-2 天级,含门禁)。

### 方案 C — 现有 IC 快路瘦身(战术)

重排/删减 _object_get_ic 快路防御:type==OBJECT 蕴含 floor、count 越界判与 capacity
合一、getter 判定移位。22 → ~14 指令。收益 ~1.3-1.5×,工作量极小。
可作为 B 的搭车项,不值单独立项。

### 方案 D — 调用点内联单体快路(排除)

编译期把 IC 快路内联到调用点:消除 call,但每访问点 ~10 指令代码膨胀
(编译器自身属性访问点数以万计 → __text 体积悬崖,违增量纪律;且汇编后 RX 页
不可自改写,无法运行期特化)。**否决。**

## 4. 推荐路线

**B → A 两段**:B 作为 v0.3 级快速交付(小、安全、独立门禁),验证收益模型;
A(形状 IC)立项为性能主线下一个里程碑,B 的站点格式正是 A 的形状缓存槽的前身
(B 证不了的形状在 A 由 shape 指针一 cmp 替代,指令框架不变)。
C 揉进 B 一并做。

**阶段切分(每段独立门禁:fixtures 362 不降 + 五目标 gen2==gen3 + macos-arm64 全链
gen1==gen2==gen3 + bench 三件套复测 + 自编译墙钟记录)**:
1. **B1**:站点槽格式 (props_ptr,index) + 快路重排(+C 瘦身);bench/prop 目标 ≤7×,自编译记录。
2. **B2**:写路径 `_object_set` 同形处理(站点缓存写目标下标);bench 复测。
3. **A 系列**(另行立项,本评审通过后出 A 的专项设计):shape 存储选型(免头字段起步)、
  追加转移表、IC 形状化、flags 交互、铁律 4 原子提交计划。

## 5. 风险登记

| 风险 | 等级 | 对策 |
|---|---|---|
| IC 快路指令重排引入分发错误(字符串/垃圾指针误判) | 中 | 自验证双 cmp(props_ptr + key 指针)不省;fixtures + test262 门禁;gen0 repro 矩阵 |
| 布局悬崖(源码体积/组合触发 miscompile) | 中 | B 属小增量;每步全链定点回放,坏窗口复测记录(本轮实测悬崖间歇活跃,见 plan.md 风险#1) |
| props_ptr 缓存与 GC/增长的交互 | 低 | 缓存仅作快路提示,cmp 失败即慢路,不影响正确性;grow 重分配自然失效 |
| A 的对象头布局变更(铁律 4) | 高(A 阶段) | A 另行专项设计,头字段变更单次提交同步全部遍历站点;优先评估免头字段 shape 存储 |

## 6. 验收指标

- bench/prop:~13× → B 后 ≤7×、A 后 ≤3×;bench/num 不退;bench/str 不相关不变。
- 自编译墙钟(出厂二进制):25.5s → B 后 ≤18s、A 后 ≤12s(记录于 CHANGELOG)。
- 门禁:fixtures 不降、五目标定点、test262 数字只升。
