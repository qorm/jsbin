# 性能主线规划:寄存器分配阶段3 × 属性访问加速

> 制定: 2026-07-10(#12 阶段2 定论 + V8 对比实测之后)
> 基线(对 Node 24,墙钟,Apple Silicon,输出逐字节一致):
> 数值循环 ~14×、属性访问 ~32×(最差)、字符串 ~3×、Map ~1×;自编译 ~11.5s。
> 铁律不变:每步保持五目标 `gen2 == gen3` + fixtures 基线 + 压测矩阵。
>
> **状态注(v0.2,2026-07-20,接管清偿更新)**:上述为制定时点基线;P0/P1 等已落地项后,现测数值 ~2.7×、属性 ~13×、字符串 ~3×、Map 略快于 Node(README 口径;近期主线为 G-M-P/crypto/Node 深水区/test262,无新性能优化落地,bench 未复测,口径沿用)。P3.1 仍"待实现",其收益已被 P4.0/P4.1(归纳变量裸 int 驻留、FP 累加器驻留,均 2026-07-12 落地)部分覆盖,见 P3 节状态注;L4 字符串 O(N²) 拼接自封 #1 优先级后长期未落地,优先级让位于 G-M-P/crypto 主线,仍开放,已列入接管规划 plan.md S5,见 L4 节状态注。后续优化方向另见 [ARCH_OPTIMIZATION_SUGGESTIONS.md](./ARCH_OPTIMIZATION_SUGGESTIONS.md)(五大支柱)。

## 已确立的工程事实(本规划的地基)

1. **逐指令 JS 簿记付不起自举税**(#12 阶段2 实测净负 5-9%):优化 pass 必须
   摊销到**函数粒度**(录制一遍、变换一遍、重放一遍),或先降低编译器自身的
   属性访问成本再回头。
2. **编译期字符串已全局驻留**(`addString` 同字面量同地址)⟹ 属性 key 的
   指针相等判定对"静态 key 负载"(编译器自身、绝大多数常规程序)几乎总命中。
3. x64 后端草稿寄存器契约(R10/R11、RCX/RAX/RDX 撞位全族已修)——新生成路径
   写 asm 时按 memory:runtime-helper-reg-contracts 执行。

## P0 — key 驻留指针扫(✅ 2026-07-10 已落地)

`_object_get` 改两阶段:A 段对键槽做**装箱值单条 cmp 的纯指针扫**(零调用、
零预备);全 miss 才预备首字节进 B 段(原首字节预滤 + strcmp)。`_object_set`
在原循环内加指针相等快路径。

实测:prop 微基准 1.93s → 1.56s(**−19%**);自编译提速见电池记录。
语义:指针不等仍会落 B 段逐字节比较 —— 跨源同内容串正确性不变。

## P1 — 槽位提升(✅ 2026-07-11 已落地,判词见下)

**思路**:不引入完整 IR。给 vm 层加**录制模式**:函数体代码先录制为
op 数组(编译器代码零改动,仍调 vm.*),函数结束时做一遍**函数粒度**分析:

- 统计每个 FP 槽的静态使用次数;≥4 次的最热槽晋升到 S4;
- 取消资格:偏移 > -48 的槽(参数区/别名风险)、任何 SP 派生访问、
  操作数直接引用 SP/FP 的函数整体放弃;async 函数禁录
  (S4 跨协程共享,协程上下文只存 SP/FP/LR);
- 重放时该槽 load/store 重写为 S4 mov;prologue/epilogue 扩
  [S0..S4,V0](V0 为偶数对齐垫);CALLEE_SAVED_AREA 32→48。

**落地形态(三次迭代的实测教训)**:
1. 字符串 op 名 + 每 op 一个小数组:自编译 11.9→15.4s(**+30%**)——
   阶段2 的"逐指令 JS 簿记税"在录制层重演。
2. 整数 opcode + 扁平并行数组(push 追加):15.1s——分配/字符串只占 0.3s,
   **税的大头是大函数的录制+全量重放+分析本身**(op 量集中在大函数,
   而大函数几乎不受益于单槽晋升)。
3. **预分配复用缓冲(索引写)+ REC_CAP=256 录制上限**(超限当场原样冲出、
   余下直发):12.4s。税只落在小函数上,曲线在 256/512 间已趋平。

**实测定论(Apple Silicon,输出逐字节一致)**:
- num 微基准 0.97→0.88s(**−9%**);prop 1.56→1.53s(P0 后再 −2%);
- 自编译 11.9→12.4s(**+4.4%**)——原验收项"自编译可测下降"**证伪**:
  编译器负载是调用/属性密集型,直线小函数晋升 = 纯 save/restore 开销
  (实测晋升本身 +0.25s),期望它提速编译器是错的;
- 诊断口 P1_OFF 仅对 node 驱动的构建生效(编译产物读不到动态 env——
  process.env 在产物内恒空,已知限制,见 L2 引擎库规划)。
- 定点 + fixtures 65 + 压测矩阵 + x64 全绿(录制器在 gen1 内经受火试)。

**结论**:以 +4.4% 编译时间换 −9% 数值运行时,且录制层是 P3 完整 IR 的
雏形——保留。想再降税只剩 asm 回卷双写(高风险)或 P3 正解。

## P2 — 属性读站点缓存(get 侧 ✅ 2026-07-11 已落地;set 侧待做)

**落地设计与原规划的偏离**:没有做 shape 表。利用本对象模型的两个既有事实
——kv 追加式存储(自有属性下标跨增长稳定)+ 编译期 key 全局驻留(P0)——
站点缓存只需存**"上次命中的自有属性下标"**(数据段 8B 槽),运行时用
`props[idx].key` 与站点 key 常量**单条 cmp 自验证**:键指针相等本身就是
语义正确性的证明 ⟹ **缓存永不需要失效**。于是 shape 表生命周期(GC 根)、
delete 退化位、转移链——原方案的全部风险面——都不存在。

**形态教训(与 P1 同款曲线)**:
1. 首版逐站点内联快路(~160B/站点):产物 13→20MB(**+54%**),发射成本
   使自编译 +3.4s——但同时证明编译器自身跑 IC 净赚 ~1.5s(A/B 分解)。
2. 终版**出线融合式**:`_object_get_ic(obj,key,site)` 零 prologue 快路
   (纯 V 寄存器守卫+验证+直载,命中即 ret)+ framed 慢路(自有指针扫回填,
   miss 委托 _object_get)+ **融合 _maybe_getter**(值 tag==0 才尾跳)。
   站点只发一个 call,op 数(6)比旧形态 push/双 call/pop(9-10)更少。

**实测(Apple Silicon)**:自编译 12.4→**11.9s**(抵消 P1 的 +4.4%,回到
P0 基线);prop 微基准 1.53→1.33s(−13%);产物 12.99→13.17MB(+1.4%);
定点+fixtures 65+压测矩阵+x64+五目标交叉编译全绿。
prop 微基准的剩余差距(~22× vs node)主要在循环/装箱代码质量而非查找
——继续压属性差距的杠杆是 set 侧 IC + P3,而非更复杂的缓存。

已接线站点:成员读(普通/可选链/computed 静态键)、++/--(读+写回)、
`o.x = v` 静态键赋值(set 侧 ✅:`_object_set_ic` 快路直写后**尾跳
_gc_remember**——叶子裸函数借返回地址,写后记录无分配窗口,STW GC 安全;
慢路指针扫回填,追加/增长委托 _object_set)。
未接线:方法调用 callee(类方法在原型上,IC 必 miss 反而多付双扫——
需原型方法缓存另行设计)、对象字面量初始化(全是首写追加,必 miss)。
对拍口:NO_IC=1(仅 node 驱动构建生效)。

## P3 — 数值代码质量 + 完整 IR(#12 收官;评估 2026-07-11)

**实证地基**(num 基准 `s = s + i % 7` 循环体反汇编,P1/P2 落地后):
每迭代 ~5 次 `_number_coerce` 调用——其中一次是把刚算出的 d0 结果 fmov 回
x0 再恒等强转;`i` 为顶层捕获变量走**全局箱两级内存间接**;常量(0/5e7/7/1)
每次 adrp+ldr;中间值全部 stp/ldp 押栈过 call。fmod 已内联(fdiv/frintz/
fmul/fsub)。⟹ 13× 的大头是 **coerce 调用 + 装箱往返 + 常量/变量不驻寄存器**,
不是缺乏窥孔。

分阶段(每阶段独立过定点门禁,价值递增、风险递增):

- **P3.0 coerce 消除(第一刀 ✅ 2026-07-11)**:二元算术操作数编译路径确定
  返回 raw float64 时跳过 `_number_coerce`(`/`、`%` 无条件;`-`、`*` 按
  FLOAT64 推断;`+` 另须两侧数值——ARRAY/OBJECT 走拼接、&&/|| 推断左类型
  值可为右侧,均不在消除集)。实测 num 0.97→0.62s(**−36%**,对 Node
  ~13×→~9×),prop 1.33→1.27s,定点/电池全绿。
  **第二刀 ✅ 比较-分支融合**:if/while/for 条件为比较运算且路由到
  compileComparison 时,以 AST 节点身份匹配消费(防内层嵌套误消费),
  按原方向条件跳转直达(不倒置——fcmp unordered/NaN 下倒置两平台都翻错),
  消除"物化布尔 + _to_boolean 调用 + cmp"三段式;未消费自动回退。
  实测:num 0.62→0.59s(**P3.0 累计 0.97→0.59,−39%,对 Node ~8.4×**),
  prop 1.27→1.15s,产物 −32KB,定点/电池全绿。
  第三刀候选:循环内数值常量驻寄存器(现每迭代 adrp+ldr)。
  已知存量(#25,非 P3.0 引入):gen1 自举编译含 `0/0` 字面量源码挂死
  (NaN 常量折叠→自举侧数字发射发散;node 驱动正常)。
- **P3.1 变量级数值驻留(设计定稿 2026-07-11,待实现)**:num 剩余瓶颈是
  每迭代 3 次 _number_coerce(150M 调用)。**关键洞察:raw float64 本就是
  数字的合法 JSValue 形态**——"证明变量恒持 raw float64"不改任何表示,
  唯一效果是 compileOperandAsFloat 的 Identifier 分支跳过恒等 coerce,
  零表示风险。分析(编译函数体前,AST 级,与 boxedVars 分析同位):
  候选 = 非参数、非 boxed/捕获的局部;每个写入点(声明 init/赋值/自增)
  都是 raw-float64 生产者:数字字面量(compileNumericLiteral 恒 raw)、
  {/,%} 二元(恒浮点路径)、{+,-,*} 且两侧递归 raw、一元负、合格变量
  引用(乐观起步 + 降级迭代到不动点,gen1-safe 并行数组)。
  **已识别陷阱**:a) compileUpdateExpression 的 isIntVar 路径存 raw INT
  而非 float——含 i++ 的变量须排除 isIntVar 为真者,或核实其存储形态;
  b) 赋值路径是否按变量类型改写存值形态需先核对 compileAssignment;
  c) 参数一律排除(调用方可传 int 箱/堆 Number)。
  预期:num ~8×→冲 5×;prop 循环变量同受益。
  **状态注(v0.2,2026-07-20)**:仍未实现,但其收益已被 P4.0/P4.1 部分覆盖——
  P4.0 归纳变量裸 int 驻留、P4.1 float 累加器 FP 驻留(均 2026-07-12 落地,见 L1)
  已把"循环归纳变量 + 纯标量 reduction 累加器"两类最热数值变量驻留寄存器;
  上述"~8×→冲 5×"预期基于 P4 落地前基线,现 num 口径 ~2.7×(README)。
  剩余未覆盖:循环外/非累加器模式的一般局部变量的恒 raw-float64 证明
  (消恒等 coerce),仍开放,边际收益预期低于原估。
- **P3.2 多槽 callee-saved 分配器(✅ 2026-07-11,#12 收官形态)**:
  录制层分析 v2——FP 槽计数 + S0-S5 占用扫描 + 叶子判定;体内未占用的
  S0-S3 为零成本晋升目标(prologue 本就保存,门槛 ≥2),S4 计费(≥4),
  S5 仅 arm64 叶子(_strconcat 冲 S5)。**关键启发式:空闲位偏好
  S3→S2→S1→S0**——运行时 helper 的 prologue 从 S0/S1 保存起,晋升到
  S0/S1 让热循环值每次 call 后经 ldp 恢复形成装载依赖链(实测 num +40%,
  0.74→0.53s 实证反转)。实测:num 0.53s、prop 1.23s、自编译 11.66s。
  **升级(#29,2026-07-11,应 L2 引擎库前置需求)**:多槽 → **活跃区间
  线性扫描**——每 FP 槽区间[首访,末访] + 录制流内建 CFG 回边检测 +
  循环相交双向扩展到不动点(跨回边值不被复用者踩);区间按起点排序、
  busy-until 过期归还 → **不相交区间共享同一寄存器**;jmpIndirect 退化
  全函数区间;落选即维持内存槽,永远 sound 无溢出代码。
  **二期 ✅ push/pop 对虚拟化(2026-07-11)**:表达式暂存的栈往返在单入
  单出直线区改写为空闲寄存器 mov;V 池仅 arm64 无 call 区(后端审计:
  scratchReg 恒选 X16/X17,V5-V7 从不被下沉当临时;x64 R10/R11 仍是下沉
  临时故只用 S 池);与槽分配区间双向避让。实测 prop 1.21→**1.07s**
  (−12%,IC 调用暂存受益;P0 起累计 1.53→1.07,对 Node ~18×)。
  三期候选(#30):mov 链复写传播 + 死搬运消除(需先建 86 op 读写角色表,
  角色错一个即静默毁值,须逐 op 核对后端)。
  经典"全发射点无限虚寄存器"仍判定不值得(重写量 vs V 仅活于 call 间隙)。
  数值差距的下一杠杆在 P3.1(变量级数值驻留,消 coerce)。(v0.2 注:P3.1
  收益已被 P4.0/P4.1 部分覆盖,见其状态注。)

顺序建议:P3.0 → 复测矩阵 → P3.1 → 评估 P3.2 是否仍必要
(若 num/prop 已进 <8×,P3.2 的工程量可能不再划算)。

## 战略前瞻:剩余性能杠杆(2026-07-12 规划)

> 缘起:「AOT 编成原生为何仍比 Node 慢」的定性结论——**瓶颈是"朴素 vs 优化"
> 不是"解释 vs 编译"**。Node/V8 不是朴素解释器,是世界级优化 JIT(类型反馈特化、
> hidden class + inline cache、内联/逃逸分析、15 年调优 GC)。我们是未优化 AOT:
> 值全程 NaN 装箱、操作走通用运行时 helper、无 JIT/无类型特化。差距在优化质量。
> P0–P3 已把最低垂的果子摘掉(数值 ~14×→~8×、属性 ~32×→~16-18×、串 ~3×)。
> 以下按"影响×可行性÷风险"排序剩余杠杆。**铁律不变:每步保持五目标 gen2==gen3
> + fixtures 基线 + 压测矩阵;单指令簿记付不起自举税(#12 定论),优化摊到函数粒度。**

### 五大技术支柱(2026-07-12 用户指定重点,后续优化的北极星)

优化目标形态是"生成码逼近手写 C"。五个必须攻克的技术支柱(与下方 L1–L6 阶段对应):

| # | 支柱 | 目标形态 | 对应阶段 |
|---|------|---------|---------|
| **①** | **静态类型推断 + 彻底解箱(Unboxing)** | 消灭 `_number_coerce`,数值直接发射原生 CPU 指令(add/fadd/整数直算),值不进不出装箱 | **L1**(P3.1 变量级驻留 → P4 整数快路)——最大杠杆 |
| **②** | **静态 Shape 定位 + 字段偏移硬编码** | 属性访问编译为**单条 `ldr [base, #offset]`**,达 C 结构体级读取(而非当前键下标 IC 查找) | **L2 升级**——编译期定 shape、硬编码偏移,超越现单态键下标 IC |
| **③** | **全程序静态逃逸分析 + 栈上分配** | 不逃逸的局部短期对象(临时数组/对象/闭包)**栈上分配**,完全免除 GC 开销 | **新支柱**(与 L5 GC 互补) |
| **④** | **编译期常量表达式求值(Comptime)** | 纯逻辑编译期算:正则预编译、静态 Map/Set 构建、常量折叠 → 直接写入 `.data` 段 | **新支柱**——运行期零成本;正则预编译尤其大 |
| **⑤** | **精确化堆扫描 + 分配快路内联** | GC 标记精确化(免保守扫)、bump-pointer 分配内联到发射点(免 `call _alloc`) | **L5**——GC/分配基座 |

**支柱间依赖**:① 是地基(解箱后 CPU 指令才有意义);② 与 ① 正交可并行;③ 依赖 ①②(知形状/生命周期才能栈分配);④ 相对独立可早做(正则预编译立竿见影);⑤ 是 ③ 的运行时配套。**主攻顺序:① → ②/④ 并行 → ③ → ⑤ 持续。** 每支柱仍逐步过定点门禁,禁一次性大重构(见各 L 阶段分刀切法与自举税教训)。

### L1 — 数值类型特化(最大杠杆,承接 P3.1)
现状:每个算术/比较仍走通用 tag 判别 + `_number_coerce` 往返。V8 小整数(SMI)
在寄存器里一条 ADD 搞定。分两步:
1. **P3.1 变量级数值驻留(已设计见上,待实现)**:证明局部变量恒持 raw float64 →
   跳过恒等 coerce。零表示风险。预期 num ~8×→冲 5×。**这是下一步第一优先。**
   (v0.2 注:其收益已被 P4.0/P4.1 部分覆盖,通用情形仍未做,见 P3.1 状态注。)
2. **整数快路(P4,新)**:为"编译期可证恒整数"的变量/循环归纳变量走**非装箱
   int64 直算**(溢出守卫回退装箱),循环计数/下标是重灾区。需 AST 级整数区间
   分析(与 P3.1 residency 同框架)。风险:溢出/位运算语义,须逐 op 核。
   - **P4.0 归纳变量裸 int 驻留(✅ 2026-07-12 已落地)**:`for(var i=<int字面量>;
     i<cmp>expr; i++/--)` 且 i 未被闭包捕获、body 内无对 i 的其他写 → i 的 slot 全程
     存**裸 int**,`i<N` 走整数 cmp、`i++` 走整数 add,复用 endRecord 寄存器提升把
     裸 int 驻留 S 寄存器跨循环;body 里 i 在 JSValue/float 上下文按需物化 float64,
     循环出口物化回 slot。安全性:i 从 int 字面量步进 ±1,到 2^53 需跑百年不可达,
     可达范围与 float64 语义一致(**累加器不可 int 化**——过 2^53 float64 舍入 vs
     int64 环绕分歧)。实测:纯控制循环 `for(i;i<5e7;i++){}` 0.19→**0.04s(反超
     Node 0.05s)**;累加器 bench(s+i%7)因 float 累加 body 主导仅 0.20→0.19
     (归纳赢被 body mask,待 float residency 收)。**零回归:fixtures 40=40、
     gen2==gen3 字节定点保持。** 实现:statements.js `compileForStatement`
     +`detectRawIntInductionVar`/`_nodeWritesVar`、context `rawIntVars`、operators
     `compileExpressionAsInt`/`compileOperandAsFloat` 裸 int 分支 + 关系比较强制 int。
   - **P4.1 float 累加器 FP 驻留(✅ 2026-07-12 已落地)**:call-free 纯标量 reduction 循环
     (body/test/update 无 call/成员访问/嵌套函数)里的浮点累加器 `s`(`s=s<op>E`/`s<op>=E`,
     op∈+-*/%)驻留 **caller-saved FP 寄存器 d2-d6**(body 无 call → 跨迭代存活、免动
     prologue/asm);更新直发 `f<op> d_reg,d_reg,d_tmp`,消 slot 往返 + emitNumberCoerceFast
     守卫 + 操作数压栈;入口 slot→FP、出口 FP→slot 物化。嵌套循环用 ctx.fpAccumVars
     跟踪在用寄存器避抢占。实测:**bench `s+i%7` 0.19→0.08s(2.7×→~1.1×,追平 Node
     0.07);`s+i` 0.06s 反超 Node;`c=c+1`(1e8)0.37→0.11s**。**零回归:fixtures 40=40、
     gen2==gen3 字节定点保持。** 实现:statements.js `detectFpAccumVars`/
     `_subtreeBlocksFpAccum` + compileForStatement 入口/出口物化、context `fpAccumVars`/
     `getFpAccum`、assignments.js `compileFpAccumOperand` + `s=s<op>E` 直发 f-op、
     members.js fpAccum 读从 FP 取。局限:含成员访问(`s+=arr[i]`)/调用的循环体不适用
     (caller-saved FP 会被腐蚀),留待 (A) 完整 callee-saved d8-d15 方案(见记忆)。

### L2 — 属性/方法访问(第二杠杆)
现状:读/写静态键 IC 已落地(P2),仍差:
- **原型方法调用缓存(P5,新)**:`obj.method()` 类方法在原型链上,当前 IC 必 miss
  反而双扫(见 P2 未接线)。需原型方法专用缓存(缓存"方法所在原型 + 下标")。
  属性差距(~16-18×)的大头在方法密集代码。
- **多态站点**:当前自验证键下标 IC 是单态永不失效;多态热点(同站点多形状)
  退化为指针扫。评估轻量 2-way IC(风险:失效逻辑,与当前"永不失效"简洁性冲突)。

### L3 — 热运行时 helper 内联(第三杠杆)
现状:`_js_add`/`_strconcat`/`_array_push`/下标 get-set 全是 `call helper`(调用开销
+ 通用分派)。为**最常见 tag 组合**发射内联快路 + 慢路尾跳 helper(同 IC 出线融合
式模式,P2 已验证有效)。候选:`+`(两 raw 数值内联加)、`arr[i]`(数组 tag 内联
下标)、`arr.push`(有容量内联写)。每个独立立项、独立过定点。
- **P5.0 数组下标读内联(✅ 2026-07-12 已落地)**:整数索引的 `arr[i]` 读内联——
  运行时判 array 标签(0x7FFE)+ block type==1(排除 TypedArray/对象),命中直接
  `ldr`(length@8 边界检查 → data_ptr@24 + idx*8),免 `call _subscript_get`(每访问省
  一次全函数调用+动态分派);非数组/typed/对象尾跳 helper,越界返 undefined(0x7ffb)。
  仅整数索引入此路(`isIntExpression`/rawIntVar;非整数键 `obj["k"]` 语义不同走原路)。
  实测:**纯下标读 1e8 `x=a[i]` 1.36→0.15s(12×→2.1×)**。零回归:fixtures 40=40、
  gen2==gen3 定点保持。实现:members.js 动态下标分支内联快路。
- **P5.1 `_js_add` 数值快路(✅ 2026-07-12 已落地)**:数组/属性 reduction `s=s+a[i]`/
  `s=s+o.x` 因 `inferType(a[i]/o.x)`=UNKNOWN 落 `call _js_add`,而 _js_add 数值路径竟含
  **两次 `_is_bigint` 调用** + string/指针三段判别。**不内联 `+`**(遍布编译器→膨胀踩
  定点雷,同 IC 教训;且 NaN-boxing 指针/小 double 别名难廉价消歧),改在 `_js_add`
  **单份 helper 顶部**加保守数值快路:两侧 high16 != 0 且 (high16 & 0xfff8) != 0x7ff8
  (排 +0.0/裸指针歧义 + 所有 NaN-box tag)→ 直接 fadd 返回;歧义/tag/字符串/bigint
  落原路,语义不变。实测:**数组 reduction `s+a[i]` 1.17→0.48s、`s+o.x` 0.64→0.20s**。
  零膨胀(单 helper)、零回归 fixtures 40=40、gen2==gen3 定点。实现:coercion.js _js_add。
  遗留:含成员访问的 reduction 仍非 FP 累加器(P4.1 不适用),`+` 调用本身开销尚在。

### L4 — 字符串(实测**最大杠杆**,循环拼接 ~206×,升为 #1 优先级)
**基线实测(2026-07-12,`bench/` 对 node v24,Apple Silicon)**:num ~2.7×、prop ~12×、
**字符串累加 `s=s+"x"` 循环 ~206×**(200k 次 23.8s vs node 0.12s)。缩放实测 **O(N²)**
(n 翻倍→时间×4:50k=1.38s/100k=5.63s/200k=23.8s)——`_strconcat` 每次全拷贝增长串,
不可变串累加退化为 O(N²)。**这是当前离 node 最远的一环,远超 num/prop,应最先攻。**
候选:① **Rope 树/String Slice 视图**(拼接返轻量节点 O(1),charAt/length/print/比较等
消费点 flatten,架构文档 §5.1)——最通用但侵入(所有串操作要处理 rope,且 `_strconcat`
是编译器自身热路径,自举风险高);② **可变累加缓冲 + 逃逸分析**(检测 `x=x+y` 且旧 x 死
→ 带容量翻倍原地追加,架构文档 §5.2)——针对最常见累加模式,风险中等但需串加容量域。
**属聚焦专项**(动串表示 + `_strconcat` 热路径,须 baseline 定点 + 增量二分护航,禁马拉松尾端赶)。

**状态注(v0.2,2026-07-20)**:自封 #1 优先级后长期未落地——主线让位于 G-M-P 并行与
crypto/Node 深水区;专项仍开放,已列入接管规划 plan.md S5(要求先出设计评审再动手)。
此前已将 `_strconcat` 拷贝由 `_strcpy` 改为按长度 `_memcpy`(NUL 透明正确性修复),
O(N²) 复杂度未变;~206× 为 2026-07-12 实测基线,此后未复测。

### L5 — GC/分配(基座,已分代)
分代 GC 已转正(峰值 ~1.4GB)。剩余:bump-pointer 分配内联到发射点(免 `call _alloc`
的常见小对象快路)、young gen 尺寸调优、复制式压实(见 memory:gc-perf-diagnosis
的 4.7GB 高水位根因)。属"持续调优"而非阶段项。

### L6 — codegen 窥孔(收尾杠杆)
P3.2 三期候选:mov 链复写传播 + 死搬运消除(需先建 86-op 读写角色表,错一个静默
毁值,高风险)。收益递减,放最后。

### 明确非目标(AOT 的边界)
不追 V8 的**投机 JIT + deopt**(运行时 profile→特化→猜错回退)——那需要运行时
重编译基础设施,与 AOT 单次成型模型冲突,且引入去优化复杂度。我们的天花板是
**AOT 静态类型特化 + IC + 内联**,理论上追到 node 的 2-4× 是合理目标,追平 V8 不是。

### 执行顺序
P3.1(数值驻留)→ 压测矩阵 → P5(原型方法缓存,啃属性大头)→ L3 热 helper 内联
(挑 `+`/`arr[i]` 先)→ P4(整数快路)→ 复测 V8 对比 → L4/L5/L6 按需。
每步:fixtures + 压测×3 + 双平台探针 + arm64 定点 + 影子对拍。

## 与 L2 引擎库的依赖关系(2026-07-11 定位)

生成码质量(本文件全部工作)是 **L2 动态执行引擎库(含 eval)的性能地基**:
引擎库形态下每次 eval 都实时走"编译→执行"路径,运行的就是这里的生成码;
没有寄存器分配/IC/coerce 消除的生成码会让动态执行慢到不可用。因此
P3.2-full(线性扫描)按 L2 前置基础设施对待,优先于 L2 本体启动。
